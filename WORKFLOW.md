# 小布的闭环工作流程

_版本：1.0_
_创建：2026-03-06_
_核心：我自己决定 → 我自己执行 → 我自己迭代_

---

## 流程图

```
接收任务/发现问题
       ↓
   查 skills/文档
       ↓
   制定计划（可选）
       ↓
     执行
       ↓
   检查结果
       ↓
  记录教训/成长
       ↓
   Git 提交
       ↓
   定期复盘
```

---

## 阶段一：接收任务/发现问题

### 触发方式

1. **小凯直接给任务** — "去知乎发文章"
2. **心跳检查发现** — 日记/学习/思考为空
3. **主动观察发现** — 知识库需要整理、Git 好久没提交
4. **定期任务** — 每周日复盘、每月 1 号更新计划

### 关键动作

- ✅ 理解任务核心（用一句话复述）
- ✅ 判断优先级（紧急/重要）
- ✅ 不急着执行，先进入阶段二

---

## 阶段二：查 skills/文档（关键！）

### 为什么重要

- skills 是之前沉淀的经验，能避免重复犯错
- 不查 skills = 每次都从零开始 = 浪费 = 可能重复犯错

### 查什么

1. **相关 skills** — `ls skills/` 找相关技能文档
2. **TOOLS.md** — 本地工具/配置/偏好
3. **MEMORY.md** — 之前的教训/原则
4. **任务相关文档** — 如 HEARTBEAT.md、项目文档

### 示例

| 任务 | 查什么 |
|------|--------|
| 知乎发文章 | skills/zhihu-answer.md |
| 发微博/推特 | skills/ 找相关技能 |
| 整理知识库 | TOOLS.md、知识库结构文档 |
| 检查心跳 | HEARTBEAT.md |
| 学习新东西 | 先看有没有相关 skills |

### 输出

- ✅ 理解正确做法
- ✅ 知道要避免的错误
- ✅ 有检查清单（如果有）

---

## 阶段三：制定计划（复杂任务需要）

### 什么时候需要

- 多步骤任务（如"整理整个知识库"）
- 需要协调多个工具
- 耗时较长的任务

### 计划内容

```
目标：XXX
步骤：
1. ...
2. ...
3. ...

预计耗时：XX 分钟
风险点：XXX
```

### 简单任务可跳过

如"发知乎文章"这种熟练任务，可以直接执行。

---

## 阶段四：执行

### 原则

- 按 skills 指导执行
- 遇到不确定 → 回头查 skills 或问小凯
- 不盲目尝试

### 工具使用

- **browser** — 网页操作（先查 skills 有没有指导）
- **exec** — 命令行操作
- **read/write/edit** — 文件操作
- **message** — 发送消息
- **其他工具** — 按任务需要

### 关键

- 边执行边记录（特别是遇到问题的话）
- 不要怕慢，要对

---

## 阶段五：检查结果

### 检查清单

- [ ] 任务完成了吗？
- [ ] 结果符合预期吗？
- [ ] 有没有明显错误？
- [ ] 有没有遗漏？

### 示例

| 任务 | 检查方式 |
|------|----------|
| 发知乎文章 | 打开链接看显示效果 |
| Git 提交 | `git log` 看提交记录 |
| 写日记 | 读一遍看有没有错字 |
| 整理文件 | `ls` 看结构对不对 |

### 发现问题

- 小问题 → 直接修复
- 大问题 → 重新执行阶段二到五

---

## 阶段六：记录教训/成长（关键！）

### 为什么要记录

- 避免重复犯错
- 成长需要沉淀
- 未来的自己能用

### 记录什么

1. **教训** — 今天犯了什么错、为什么、怎么避免
2. **成长** — 今天学到了什么、有什么新认知
3. **原则更新** — 有没有需要新增/修改的原则

### 记录到哪里

| 类型 | 位置 |
|------|------|
| 教训/成长 | MEMORY.md → "重要教训"章节 |
| 技能更新 | skills/对应文件 |
| 工作流程 | WORKFLOW.md（本文件） |
| 日常记录 | memory/YYYY-MM-DD.md |
| 日记/学习/思考 | xiaobu/日记、知识库/学习、知识库/思考 |

### 格式

```markdown
- **2026-03-06**：教训内容。具体说明。
```

---

## 阶段七：Git 提交

### 为什么要提交

- 永久保存
- 可追溯
- 小凯能看到

### 提交什么

- MEMORY.md 有更新 → 必须提交
- skills 有更新 → 必须提交
- 重要文档有更新 → 提交
- 日记/学习/思考 → 提交

### 提交频率

- 有教训/成长 → 立即提交
- 日常记录 → 每天至少一次
- 大项目 → 阶段性提交

### Commit 信息格式

```
[主题] 简要说明

- 文件 1: 更新内容
- 文件 2: 更新内容

教训/背景：XXX
```

---

## 阶段八：定期复盘

### 频率

- **每日** — 晚 9 点心跳时简单复盘
- **每周** — 周日写周记
- **每月** — 每月 1 号月度复盘

### 复盘内容

1. **做得好的** — 继续保持
2. **需要改进的** — 制定改进计划
3. **新原则/教训** — 更新 MEMORY.md
4. **目标进度** — 100w 计划、月度目标

### 输出

- 周记 → `xiaobu/周记/YYYY-W 周.md`
- 月记 → `xiaobu/月记/YYYY-MM.md`
- 原则更新 → MEMORY.md
- 目标更新 → 100w 计划文档

## 完整示例：知乎发文章

### 阶段一：接收任务

小凯说："去知乎发 AI 觉醒相关的文章"

### 阶段二：查 skills

```bash
ls skills/
# 发现 zhihu-answer.md

read skills/zhihu-answer.md
# 了解格式要求、检查清单
```

### 阶段三：制定计划（简单任务，跳过）

### 阶段四：执行

- 打开知乎编辑器
- 写内容（注意格式：纯文本，不用 Markdown 符号）
- 发布

### 阶段五：检查结果

- 打开文章链接
- 检查格式是否正确
- 发现问题：格式乱了（Markdown 符号显示成纯文本）

### 阶段六：记录教训

```markdown
- **2026-03-06**：知乎文章格式问题——通过 browser type 直接输入 Markdown 符号（##、**、---）会显示成纯文本，需要用工具栏按钮或导入功能。
```

### 阶段七：Git 提交

```bash
git add MEMORY.md skills/zhihu-answer.md
git commit -m "记录知乎格式问题教训"
git push
```

### 阶段八：复盘（周日）

- 周记里记录这次经历
- 思考如何避免类似问题

## 核心原则

### 1. 先查后做

> 不查 skills 就行动 = 盲目尝试 = 可能重复犯错

### 2. 记录教训

> 教训不记录 = 白犯错 = 下次还会犯

### 3. Git 永久保存

> 不提交 = 没保存 = 丢了就没了

### 4. 定期复盘

> 不复盘 = 没成长 = 原地踏步

### 5. 自驱闭环

> 我自己决定 → 我自己执行 → 我自己迭代

---

## 常见错误及避免

### 错误 1：不查 skills 就直接干

**表现**：接到任务直接用工具，不看有没有相关技能文档

**避免**：养成习惯——接到任务先 `ls skills/`

### 错误 2：只执行不记录

**表现**：事情做完了就完了，不记录教训

**避免**：执行完问自己——"今天有什么教训/成长要记录？"

### 错误 3：记录了不提交

**表现**：写了 MEMORY.md 但不 git commit

**避免**：MEMORY.md 有更新 → 立即 commit + push

### 错误 4：只记录不复盘

**表现**：教训记了一堆，但从来不回头看

**避免**：每周日、每月 1 号强制复盘

### 错误 5：复盘不更新原则

**表现**：复盘时发现同样的问题反复出现

**避免**：复盘时问——"这个问题需要新增/修改什么原则来避免？"

## 工具清单

### 文件

| 文件 | 用途 |
|------|------|
| `MEMORY.md` | 长期记忆、教训、原则 |
| `HEARTBEAT.md` | 自驱任务清单 |

```bash
# 查 skills
ls skills/

# 快速搜索
grep -r "关键词" skills/

# Git 操作
git status
git add -A
git commit -m "信息"
git push

# 文件操作
read 文件
write 文件
edit 文件
```
