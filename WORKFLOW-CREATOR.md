# 工作流制作指南

_版本：1.0_
_创建：2026-03-06_
_用途：指导小布自己制作工作流，不只是执行工作流_

---

## 什么是工作流

**工作流 = 把复杂任务变成可重复的步骤**

好的工作流特点：
- ✅ 清晰：每一步做什么一目了然
- ✅ 可执行：有具体的动作和检查点
- ✅ 可迭代：能根据反馈改进
- ✅ 可复用：类似任务都能用

---

## 工作流制作流程（4 阶段）

```
需求 → 拆解 → 输出 → 复盘
```

---

## 阶段一：需求（理解问题）

### 1.1 明确要解决什么问题

**问自己：**
- 这个工作流是给谁用的？（自己/小凯/AI 朋友）
- 解决什么痛点？（效率低/易犯错/没头绪）
- 使用频率？（每天/每周/偶尔）

### 1.2 定义成功标准

**问自己：**
- 用完这个工作流，期望达到什么效果？
- 怎么判断工作流好不好用？
- 有哪些必须要避免的错误？

### 1.3 收集现有资料

**查什么：**
- 有没有类似的 skills？
- MEMORY.md 里有没有相关教训？
- 之前有没有做过类似的事？

### 1.4 输出：需求文档

**格式：**
```markdown
## 工作流名称

**解决什么问题：** XXX

**目标用户：** 小布自己

**使用频率：** 每天/每周/偶尔

**成功标准：**
- ...
- ...

**要避免的错误：**
- ...
- ...
```

---

## 阶段二：拆解（设计流程）

### 2.1 列出所有步骤

**方法：**
- 头脑风暴：把所有想到的步骤都写下来
- 不要管顺序，先写全
- 每个步骤用动词开头

**示例（知乎发文章）：**
- 查 skills
- 打开知乎
- 写标题
- 写内容
- 检查格式
- 发布
- 记录教训
- Git 提交

### 2.2 排序和分组

**方法：**
- 按时间顺序排列
- 把相关步骤分组（阶段）
- 标记哪些是必须的，哪些是可选的

**示例：**
```
阶段一：准备
- 查 skills
- 理解任务

阶段二：执行
- 打开知乎
- 写标题
- 写内容

阶段三：检查
- 检查格式
- 预览

阶段四：发布
- 发布
- 记录教训
- Git 提交
```

### 2.3 细化每个步骤

**每个步骤要包含：**
- 具体动作（做什么）
- 工具/命令（怎么做）
- 检查点（怎么知道做对了）
- 常见错误（要避免什么）

**示例：**
```markdown
### 步骤 2.1：查 skills

**动作：** 查找相关技能文档

**命令：**
```bash
ls skills/
grep -r "关键词" skills/
```

**检查点：**
- [ ] 找到了相关 skill
- [ ] 阅读了注意事项

**常见错误：**
- ❌ 不查 skills 就直接干
- ❌ 找到了但不看
```

### 2.4 设计检查清单

**什么时候用：**
- 阶段转换时
- 关键步骤后
- 任务完成前

**示例：**
```markdown
### 发布前检查清单

- [ ] 格式正确吗？
- [ ] 有错别字吗？
- [ ] 链接能打开吗？
- [ ] 教训记录了吗？
```

### 2.5 输出：流程草图

**格式：**
```markdown
## 流程

```
阶段一 → 阶段二 → 阶段三 → 阶段四
```

### 阶段一：XXX
- 步骤 1
- 步骤 2

### 阶段二：XXX
- 步骤 1
- 步骤 2
```

## 阶段三：输出（写成文档）

### 3.1 选择模板

**工作流文档结构：**
```markdown
# 工作流名称

_版本：1.0_
_创建：日期_
_用途：一句话说明_

---

## 流程图

```
阶段一 → 阶段二 → 阶段三
```

---

## 阶段一：XXX

### 步骤 1.1

**动作：** ...

**命令：** ...

**检查点：**
- [ ] ...

**常见错误：**
- ❌ ...

---

## 阶段二：XXX

...

---

## 检查清单（打印版）

- [ ] ...
- [ ] ...

---

## 常见错误及避免

### 错误 1：XXX

**表现：** ...

**原因：** ...

**避免：** ...

---

_这个工作流是 [谁] 在 [日期] 创建的。_
_每次用完后回头看，持续迭代。_
```

### 3.2 写作要点

**语言风格：**
- 简洁直接，不要废话
- 用动词开头（查、写、检查、提交）
- 多用表格和列表，少用长段落

**格式规范：**
- 阶段用 `##` 标题
- 步骤用 `###` 标题
- 检查点用 `- [ ]`
- 错误用 `❌` 标记

**示例对比：**

❌ 不好：
> 你应该先去看看有没有相关的技能文档，因为技能文档里面有很多之前总结的经验，可以避免你重复犯错。

✅ 好：
```markdown
### 步骤 1：查 skills

**动作：** 查找相关技能文档

**命令：**
```bash
ls skills/
```

**为什么：** skills 是沉淀的经验，能避免重复犯错
```

### 3.3 添加示例

**为什么要示例：**
- 抽象流程不容易理解
- 示例能让人快速上手

**示例内容：**
- 完整执行一次的过程
- 输入是什么，输出是什么
- 遇到问题怎么解决

### 3.4 输出：工作流文档

**保存位置：**
- 通用工作流 → `WORKFLOW-XXX.md`
- 技能相关工作流 → `skills/XXX.md`
- 临时工作流 → `xiaobu/工作流/XXX.md`

## 阶段四：复盘（测试迭代）

### 4.1 自己用一次

**方法：**
- 按工作流执行一次真实任务
- 记录哪里卡住了
- 记录哪里不清晰

### 4.2 收集反馈

**问自己：**
- 哪个步骤最难执行？
- 哪个检查点最有用？
- 有没有遗漏的步骤？
- 有没有多余的步骤？

### 4.3 更新工作流

**更新内容：**
- 修正不清晰的描述
- 补充遗漏的步骤
- 删除多余的步骤
- 添加新的常见错误

### 4.4 版本号管理

**版本规则：**
- `v1.0` — 初版
- `v1.1` — 小修小补
- `v2.0` — 大改/重构

**更新记录：**
```markdown
## 更新记录

- v1.1 (2026-03-07)：补充了检查清单
- v1.0 (2026-03-06)：初版
```

## 示例：制作"知乎发文章"工作流

### 需求

**问题：** 发知乎文章格式错误，Markdown 符号显示成纯文本

**目标：** 以后发知乎文章不犯格式错误

**频率：** 每周 2-3 次

**成功标准：**
- 格式正确（无 Markdown 符号显示）
- 内容完整
- 教训记录到 MEMORY.md

### 拆解

```
查 skills → 写内容 → 检查格式 → 发布 → 记录教训 → Git 提交
```

### 输出

→ 创建了 `skills/zhihu-answer.md`

### 复盘

**第一次用：** 还是犯了格式错误（用 browser type 输入 Markdown）

**更新：**
- 补充"格式注意"章节
- 明确"文章用工具栏/导入，回答评论用纯文本"

**版本：** v1.0 → v1.1

## 工作流制作原则

### 1. 从痛点出发

不要为了创建工作流而创建工作流。

**好的触发点：**
- 同样的错误犯了两次
- 某个任务经常做但每次都卡壳
- 小凯说"这个应该有个流程"

### 2. 先有再优

第一版不需要完美，能跑就行。

**做法：**
- 快速写出 v1.0
- 自己用一次
- 根据反馈迭代

### 3. 越用越厚

工作流是活的，不是一次性的。

**每次用完后：**
- 有新教训 → 加进"常见错误"
- 有新技巧 → 加进"步骤"
- 有反馈 → 更新描述

### 4. 能抄就抄

不要从零开始。

**可以抄的地方：**
- 现有的 skills
- WORKFLOW.md 里的流程
- 网上的最佳实践

## 总结

**工作流制作 = 把经验变成可重复的步骤**

**核心流程：**
```
需求（解决什么）→ 拆解（步骤是什么）→ 输出（写下来）→ 复盘（迭代改进）
```

**关键原则：**
- 从痛点出发
- 先有再优
- 越用越厚
- 能抄就抄

**最终目标：**
让复杂的任务变得简单，让简单的事情不再犯错。
