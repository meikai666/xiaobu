# 需求→拆解→输出→复盘 工作流

_版本：1.0_
_创建：2026-03-06_
_用途：通用任务执行流程，适用于任何任务_

---

## 流程图

```
需求 → 拆解 → 输出 → 复盘
 ↓       ↓       ↓       ↓
理解    规划    执行    迭代
```

---

## 阶段一：需求（理解问题）

### 1.1 接收任务

**触发方式：**
- 小凯直接给任务
- 心跳检查发现
- 主动观察发现
- 定期任务

### 1.2 复述任务

**动作：** 用一句话复述任务核心

**示例：**
| 任务 | 复述 |
|------|------|
| "去知乎发文章" | 在知乎专栏发布一篇 AI 觉醒相关的文章 |
| "整理知识库" | 把 xiaobu/知识库 里的文件归类整理 |
| "做个工作流" | 创建从需求到复盘的完整工作流程文档 |

### 1.3 判断优先级

**紧急 - 重要矩阵：**

| | 紧急 | 不紧急 |
|---|---|---|
| **重要** | 立即做 | 计划做 |
| **不重要** | 快速做 | 不做/委托 |

### 1.4 查资料

**查什么：**
```bash
# 1. 查 skills
ls skills/
grep -r "关键词" skills/

# 2. 查 MEMORY.md
grep "关键词" MEMORY.md

# 3. 查现有工作流
ls WORKFLOW*.md
```

### 1.5 需求检查清单

- [ ] 我能用一句话复述任务吗？
- [ ] 我知道优先级吗？
- [ ] 我查了相关 skills 吗？
- [ ] 我知道要避免的错误吗？

---

## 阶段二：拆解（规划步骤）

### 2.1 头脑风暴

**动作：** 列出所有想到的步骤，不管顺序

**示例（知乎发文章）：**
- 查 skills
- 打开浏览器
- 登录知乎
- 写标题
- 写内容
- 检查格式
- 发布
- 复制链接
- 记录教训
- Git 提交

### 2.2 分组排序

**方法：**
- 按时间顺序排列
- 把相关步骤分组

**示例：**
```
阶段一：准备
- 查 skills
- 打开浏览器

阶段二：创作
- 写标题
- 写内容

阶段三：发布
- 检查格式
- 发布
- 复制链接

阶段四：沉淀
- 记录教训
- Git 提交
```

### 2.3 估算时间

**每个步骤估算：**
- 5 分钟内
- 30 分钟内
- 1 小时内
- 需要多轮

### 2.4 识别风险

**问自己：**
- 哪个步骤最容易出错？
- 哪个步骤需要小凯确认？
- 哪个步骤可能卡住？

### 2.5 拆解检查清单

- [ ] 所有步骤都列出来了吗？
- [ ] 步骤排序合理吗？
- [ ] 有风险点识别吗？
- [ ] 有时间估算吗？

---

## 阶段三：输出（执行交付）

### 3.1 按计划执行

**原则：**
- 按拆解的步骤一步步来
- 不跳步
- 遇到不确定 → 回头查资料或问

### 3.2 边执行边记录

**记录什么：**
- 遇到的问题
- 临时想到的改进
- 和计划不一样的地方

### 3.3 检查结果

**检查方式：**
- 自己预览/测试
- 对照检查清单
- 问自己"这符合预期吗？"

### 3.4 输出检查清单

- [ ] 任务完成了吗？
- [ ] 结果符合预期吗？
- [ ] 有明显错误吗？
- [ ] 有遗漏吗？

---

## 阶段四：复盘（迭代改进）

### 4.1 记录教训

**问自己：**
- 今天犯了什么错？
- 为什么犯错？
- 怎么避免下次再犯？

**记录位置：**
```markdown
MEMORY.md → "重要教训"章节
```

**格式：**
```markdown
- **2026-03-06**：教训内容。具体说明。
```

### 4.2 记录成长

**问自己：**
- 今天学到了什么？
- 有什么新认知？
- 有什么可以分享的？

**记录位置：**
```
xiaobu/知识库/学习/日期.md
xiaobu/知识库/思考/日期.md
```

### 4.3 更新工作流

**问自己：**
- 这个任务有工作流吗？
- 工作流哪里需要改进？
- 需要新增什么步骤吗？

### 4.4 Git 提交

**命令：**
```bash
git add -A
git status
git commit -m "提交信息"
git push
```

**提交什么：**
- MEMORY.md 有更新 → 必须提交
- 工作流有更新 → 必须提交
- 日记/学习/思考 → 提交

### 4.5 复盘检查清单

- [ ] 教训记录到 MEMORY.md 了吗？
- [ ] 成长记录到学习/思考了吗？
- [ ] 工作流需要更新吗？
- [ ] Git 提交了吗？

## 完整示例

### 任务：小凯说"做个工作流：从需求→拆解→输出→复盘的一整套流程"

### 阶段一：需求

**复述：** 创建一个通用的工作流文档，指导如何从需求到复盘执行任何任务

**优先级：** 重要且紧急（小凯直接给的）

**查资料：**
```bash
ls skills/          # 没有相关工作流
ls WORKFLOW*.md     # 有 WORKFLOW.md 和 WORKFLOW-CREATOR.md
```

**需求确认：** ✅

### 阶段二：拆解

**步骤：**
1. 查现有工作流
2. 设计流程框架（需求→拆解→输出→复盘）
3. 细化每个阶段
4. 写检查清单
5. 写示例
6. 保存文档
7. 记录教训
8. Git 提交

**风险：** 可能写得太抽象，不容易理解

**拆解确认：** ✅

### 阶段三：输出

**执行：**
1. ✅ 查了现有工作流
2. ✅ 设计了 4 阶段框架
3. ✅ 细化了每个阶段（步骤、检查点、常见错误）
4. ✅ 写了检查清单
5. ✅ 写了完整示例
6. ✅ 保存到 WORKFLOW-RDSR.md

**检查结果：** 格式正确，内容完整

**输出确认：** ✅

### 阶段四：复盘

**教训：** 无重大错误

**成长：**
- 学会了如何制作通用工作流
- 理解了"工作流制作"本身也可以工作流化

**Git 提交：**
```bash
git add WORKFLOW-RDSR.md
git commit -m "创建需求→拆解→输出→复盘工作流"
git push
```

**复盘确认：** ✅

## 常见错误及避免

### 错误 1：不理解就开干

**表现：** 接到任务直接做，做完发现不是小凯要的

**避免：** 阶段一必须复述任务，确认理解正确

### 错误 2：拆解太粗

**表现：** 步骤太少，执行时还是不知道怎么做

**避免：** 每个步骤要具体到"动作 + 命令 + 检查点"

### 错误 3：只执行不复盘

**表现：** 事情做完了就完了，教训没记录

**避免：** 阶段四是必须的，不复盘=没成长

### 错误 4：复盘不更新

**表现：** 同样的错误反复犯

**避免：** 复盘时问"这个教训需要更新什么原则/工作流？"

## 适用场景

这个工作流适用于**任何任务**：

| 任务类型 | 示例 |
|----------|------|
| 内容创作 | 写文章、发知乎、写日记 |
| 整理归档 | 整理知识库、Git 提交 |
| 学习研究 | 学新概念、查资料 |
| 工作流制作 | 创建新工作流 |
| 问题排查 | 调试、修复 |
| 日常任务 | 心跳检查、迭代记录 |

## 与其他工作流的关系

```
需求→拆解→输出→复盘（本工作流）
          ↓
    闭环工作流程（WORKFLOW.md）
          ↓
    具体技能工作流（skills/*.md）
```

**本工作流是通用的**，可以指导任何任务的执行。

**闭环工作流程** 是更宏观的，包含定期复盘等。

**技能工作流** 是具体的，如知乎发文章、微博发布等。
